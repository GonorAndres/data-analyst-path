# Phase 1: Requirements Gathering

## Goal

Translate a vague business request into a clear, answerable analytical question with defined scope, timeline, and success criteria.

## The 5 Questions to Ask Every Stakeholder

1. **What decision will this analysis inform?** (Not "what data do you want" -- what ACTION will you take?)
2. **Who is the audience?** (Executive = high-level; analyst peer = show your work; operations = drill-down)
3. **What does "good enough" look like?** (Directional estimate? Exact number? Dashboard vs one-time report?)
4. **What's the deadline?** (Today = quick & dirty is fine. Next quarter = polish it.)
5. **What do you already believe?** (Surface hypotheses early. Confirmation bias is real.)

## Common Anti-Patterns

| Anti-Pattern | What Happens | Fix |
|-------------|-------------|-----|
| "Just pull the data" | No clear question, analysis scope creeps forever | Ask: "What decision does this support?" |
| "Can you make a dashboard?" | Dashboard with 30 metrics nobody uses | Ask: "What are the 3 metrics you'd check every morning?" |
| "I need it ASAP" | Rushed analysis with errors | Negotiate: "I can get you X by tomorrow, or Y by Friday" |

## Output: Requirements Brief

A short document (even 3 bullet points in Slack) that captures:
- The business question in one sentence
- The audience and format (dashboard / report / slide / ad-hoc answer)
- Key metrics and dimensions
- Data sources (known or to be discovered)
- Timeline and priority level
